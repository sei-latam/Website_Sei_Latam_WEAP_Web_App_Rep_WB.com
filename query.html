Consultas, filtros y análisis espaciales/temporales

Paneles de consultas, descargas y visualización

1) Selección de Mapas Base.

Inputs o parámetros: Librería y repositorio SnazzyMaps en Google Maps JavaScript API para la creación de los mapas base “iconChange”, “baseChange” y “Multibrand”.

Acciones e instrucciones:

  • Configurar y definir cada uno de los mapas base, buscando personalizar y modificar el estilo visual utilizando los widgets ui.Buttom () y la función onClick (activación o desactivación del mapa base).
  • Mapa base iconChange o alta vegetación: Estilo minimalista con alto contraste, que invierte la luminosidad en el mapa y reduce el brillo general. Además, oculta determinados iconos (puntos de interés, etiquetas, lugares administrativos, etc.).
  • Mapa base baseChange o baja vegetación: Estilo con alto énfasis en elementos naturales (ríos, lagos, lagunas, montañas, cobertura vegetal, etc), invirtiendo la luminosidad y ocultando los mismos iconos que el iconChange
  • • Mapa base Multibrand o alto contraste: Estilo detallado y sofisticado, ideal para branding o visualizaciones temáticas, con etiquetas visibles y colores personalizados (carreteras, paisajes y agua en tonos oscuros). Asimismo, un alto contraste entre texto y fondo.

Unión de la información temporal y espacial.

La unión de la información temporal (series de tiempo) y espacial (área de estudio), se realizó mediante la aplicación de Sistemas de Información Geográfica (SIG) específicamente el software licenciado ArcGIS, utilizando la herramienta ‘spatial join’ para cada una de las cinco variables definidas en el proyecto, obteniendo de esta forma archivos (.shp) y una primera estructuración de los datos.

Creación de imágenes multibanda.

En una primera instancia, se optó por realizar la exportación de los archivos (.shp) en la plataforma de la nube Google Earth Engine (GEE). Sin embargo, debido a limitaciones establecidas en GEE referentes al tamaño (hasta 2 GB), numero de columnas (hasta 1000) número máximo de vértices por geometría (hasta 100.000 vértices), fue decidido convertir los archivos (.shp) a imágenes multibanda tipo ráster (.tiff o. geotiff) que ocuparan menor espacio y tuvieran una mayor eficiencia en el desarrollo de la aplicación en GEE. De esta forma, se logró importar de forma exitosa las cinco variables que contenían mediciones mensuales de las series temporales entre 1980 a 2020, en donde cada mes de las series fueron remplazadas y enumeradas como bandas ‘b1, b2, b3…. b492’ en las imágenes multibanda.

Creación de image collections mensuales y anuales.

Una vez importadas las imágenes multibanda en GEE, el siguiente paso fue convertir cada una de las imágenes multibanda a estructuras propias de GEE (FeatureCollection, ImageCollection, List, Arrays, etc). En este caso, la estructura escogida fue ImageCollection debido a su capacidad de almacenar y organizar imágenes ráster para diferentes periodos de tiempo (en este caso mensual). Asimismo, es importante mencionar que para el caso de las ImageCollections mensuales, se procedió a renombrar todas las bandas de acuerdo con la variable que pertenece ‘temperatura’, ‘precipitación’, ‘escorrentía’, ‘evapotranspiración’, entre otras.

Una vez creadas las ImageCollections mensuales, se crearon ImageCollections anuales tomando en cuenta las siguientes consideraciones por variable: En la variable de temperatura, se calculó la media anual en unidades Celsius (°). Mientras que, para las variables de precipitación, escorrentía, evapotranspiración potencial y real, fueron calculadas las acumulaciones o sumatorias anuales en unidades de milímetros (mm).

Desarrollo Front-End y Back-End

Componente Front-End.

El componente Front-End utilizado en el desarrollo de aplicaciones web, incorpora la parte visible o arquetipo final, por la cual los usuarios consiguen interactuar, consultar y filtrar de forma intuitiva o simple los datos procesados internamente en el componente BACK-END. En ese orden de ideas, fueron colocados diferentes elementos, interfaces y paneles para la realización de consultas temporales, espaciales y descarga de imágenes.

Componente Back-End.

El componente Back-End también conocido como componente o fase oculta dentro del desarrollo de aplicaciones web, es quizás la componente más importante y relevante de la aplicación WEAP_WebAppRepWB, ya que durante esta fase de construcción son realizadas las siguientes actividades:

  • Importe y acceso a las ImageCollections mensuales y anuales
  • Preprocesamiento y automatización de las imágenes
  • Creación y configuración de las interfaces, elementos y visualizadores gráficos del componente Front-End
  • Importe de librerías y extensiones externas para la creación de mapas base y simbologías
  • Desarrollo de las diferentes consultas, filtros y análisis definidas en el diseño de la aplicación web
  • Escritura y desarrollo de las diferentes versiones de código utilizadas a través de la API (Interfaz de Programación de Aplicaciones) en Google Earth Engine.

En el caso de la API de Google Earth Engine, esta integra de forma predefinida una serie de accesorios o widgets atraves de interfaces de usuario (UI) , los cuales fueron utilizados en su mayoría y son descritos brevemente a continuación.

  • Mapas ui.map(): Widget interactivo que permite adicionar, modificar o actualizar capas (layouts) de forma dinámica, generalmente a partir de otros widgets como funciones, selección o botones.
  • Paneles ui.panel(): Widget interactivo que permite adicionar múltiples widgets (botones, funciones, textos, etc.) dentro de un mismo panel, otorgando una mejor distribución en la visualización gráfica de la aplicación.
  • Botones ui.buttom(): Widget dinámico que permite activar o desactivar a partir de un ‘click’ otros widgets y funcionalidades dentro de la aplicación.
  • Selección variables ui.select(): Widget dinámico que permite seleccionar una o más variables, para después ser enviadas como parámetros de entrada o inputs a otras funcionalidades o widgets.
  • Gráficos ui.chart(): Widget dinámico que es activado generalmente dentro de una función, dentro de sus especificaciones requiere la inserción de parámetros tales como: colecciones de variables o capas, escala, tipo de gráfico (líneas, columnas, barras, etc.).
  • Títulos ui.label(): Widget estático o fijo que se utiliza para imprimir o mostrar textos, frases o instrucciones dentro de la aplicación.
  • Textos ui.Textbox(): Widget dinámico que recibe únicamente datos tipo string, text o char, entre otros. Su principal uso radica en la inserción de datos tipo fecha ‘YYYY-MM-DD’ o de caracteres.
  • Reset widgets ui.root.widgets(): Widget que se utiliza generalmente como controlador de todos los widgets utilizados en la aplicación, permitiendo por ejemplo un restablecimiento de la aplicación, así como la ‘eliminación’ de widgets activados previamente por el usuario.
